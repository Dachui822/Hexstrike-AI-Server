> **版本**: v1.0.0

## 🗄️ 存储分层策略

| 存储层 | 职责 | 数据生命周期 |
|--------|------|--------------|
| **MySQL** | 任务历史、工具元数据、审计日志、健康快照 | 持久化，支持分页/筛选/导出 |
| **Redis** | 任务队列、运行时状态、进度推送、实时日志流、分布式锁 | 运行时内存态，支持 TTL 自动清理 |

## 📊 MySQL 数据表设计

### 1. `tasks` (任务核心表)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `VARCHAR(36)` | `PK` | 任务唯一标识 (UUID) |
| `tool_name` | `VARCHAR(64)` | `NOT NULL` | 工具名称 (如 `nmap`, `sqlmap`) |
| `target` | `VARCHAR(255)` | `NOT NULL` | 攻击目标 (IP/URL/文件路径) |
| `params` | `JSON` | `DEFAULT NULL` | 工具执行参数 (由决策引擎优化后注入) |
| `status` | `ENUM('PENDING','RUNNING','SUCCESS','FAILED','CANCELLED','TIMEOUT')` | `DEFAULT 'PENDING'` | 任务状态机 |
| `progress` | `FLOAT` | `DEFAULT 0.0` | 进度百分比 (0.0 ~ 100.0) |
| `priority` | `INT` | `DEFAULT 0` | 执行优先级 (数值越大越优先) |
| `worker_id` | `VARCHAR(64)` | `DEFAULT NULL` | 分配的执行 Worker 标识 |
| `mcp_request_id` | `VARCHAR(64)` | `DEFAULT NULL` | 关联 MCP JSON-RPC 请求的 `id` |
| `started_at` | `DATETIME` | `DEFAULT NULL` | 实际开始时间 |
| `completed_at` | `DATETIME` | `DEFAULT NULL` | 结束时间 |
| `created_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `updated_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP ON UPDATE` | 最后更新时间 |
| `error_message` | `TEXT` | `DEFAULT NULL` | 失败/超时错误详情 |
| `output_path` | `VARCHAR(512)` | `DEFAULT NULL` | 结果文件/报告存储路径 |

**索引建议：**
- `idx_status_created`: `(status, created_at DESC)` → 支撑 UI 任务池分页筛选
- `idx_tool`: `(tool_name)` → 支撑工具历史统计
- `idx_mcp_req`: `(mcp_request_id)` → 支撑 MCP 异步回调追踪

---

### 2. `task_logs` (任务日志表)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 日志自增主键 |
| `task_id` | `VARCHAR(36)` | `FK -> tasks.id` | 所属任务 |
| `level` | `ENUM('DEBUG','INFO','WARNING','ERROR','CRITICAL')` | `DEFAULT 'INFO'` | 日志级别 |
| `message` | `TEXT` | `NOT NULL` | 日志内容 (含 ANSI 转义符保留) |
| `source` | `ENUM('stdout','stderr','system','progress')` | `DEFAULT 'system'` | 日志来源 |
| `timestamp` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 产生时间 |

**索引建议：**
- `idx_task_time`: `(task_id, timestamp ASC)` → 支撑 SSE 日志流顺序读取
- `idx_level`: `(level)` → 支撑日志级别过滤

### 3. `tools` (工具元数据与健康状态表)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `name` | `VARCHAR(64)` | `PK` | 工具命令名 (如 `gobuster`) |
| `display_name` | `VARCHAR(128)` | `NOT NULL` | UI 展示名称 |
| `category` | `VARCHAR(32)` | `DEFAULT 'utility'` | 分类 (`recon`, `exploitation`, `scanning`, `utility`) |
| `description` | `TEXT` | `DEFAULT NULL` | 工具用途说明 |
| `command_template` | `TEXT` | `DEFAULT NULL` | 基础命令模板 (含占位符) |
| `dependencies` | `JSON` | `DEFAULT NULL` | 依赖清单 `{"apt": ["nmap"], "pip": ["requests"]}` |
| `health_check_cmd` | `VARCHAR(255)` | `DEFAULT NULL` | 健康检测命令 (如 `nmap --version`) |
| `is_available` | `BOOLEAN` | `DEFAULT FALSE` | 当前是否可用 |
| `last_health_check` | `DATETIME` | `DEFAULT NULL` | 上次检测时间 |
| `installed_version` | `VARCHAR(32)` | `DEFAULT NULL` | 当前安装版本 |
| `created_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 注册/发现时间 |
| `updated_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP ON UPDATE` | 状态更新时间 |

### 4. `audit_logs` (操作审计表)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `BIGINT` | `PK AUTO_INCREMENT` | 审计记录 ID |

| `details` | `JSON` | `DEFAULT NULL` | 附加上下文 |
| `created_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 操作时间 |

## 🔑 Redis 键值设计对照表

MySQL 负责持久化，Redis 负责**高并发运行时状态**。以下是核心键映射：

| 键模式 | 类型 | 用途 | TTL |
|--------|------|------|-----|
| `task:{task_id}` | `HASH` | 运行时状态快照 (status, progress, worker_id) | 任务完成后 24h 清理 |
| `task:queue` | `ZSET` | 优先级任务队列 (score = priority + created_at) | 永久 |
| `task:{task_id}:logs` | `LIST` | 实时日志缓冲 (最新 1000 条) | 任务完成后 2h 清理 |
| `task:{task_id}:lock` | `STRING` | 分布式执行锁 (防止重复调度) | 30s (自动续期) |
| `tool:{name}:health` | `HASH` | 工具健康状态缓存 (is_available, version, last_check) | 10min |
| `hexstrike:progress` | `PUB/SUB` | 进度推送频道 | 无 |
| `hexstrike:logs` | `PUB/SUB` | 实时日志推送频道 | 无 |

## 🔗 与 MCP 协议的映射关系

### 1. MCP Client 调用示例
```json
{
  "jsonrpc": "2.0",
  "id": "req-12345",
  "method": "tools/call",
  "params": {
    "name": "nmap",
    "arguments": {
      "target": "192.168.1.100",
      "scan_type": "-sV -sC",
      "ports": "80,443,8080"
    }
  }
}
```

### 2. 后端处理流程
1. 生成 `task_id = uuid4()`
2. 写入 MySQL `tasks` 表 (`status=PENDING`, `mcp_request_id="req-12345"`)
3. 写入 Redis `task:{task_id}` + `task:queue`
4. 返回 MCP 响应:
```json
{
  "jsonrpc": "2.0",
  "id": "req-12345",
  "result": { "task_id": "<task_id>", "status": "ACCEPTED" }
}
```
5. 执行过程中通过 `hexstrike:progress` 推送进度，通过 `hexstrike:logs` 推送日志
6. 完成时更新 MySQL `tasks.status` & `task_logs`，清理 Redis 临时键
